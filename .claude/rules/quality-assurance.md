# 품질 보증 규칙

> 에이전트 출력물의 품질 평가, 피드백 루프, 지속적 개선을 위한 기준

---

## 1. 에이전트 출력 품질 평가 기준

### 1.1 공통 품질 체크리스트

모든 에이전트 출력물에 적용되는 최소 품질 기준:

| # | 항목 | 기준 | 필수 |
|---|------|------|------|
| 1 | **형식 준수** | reporting-standards.md 형식에 맞는가 | ✅ |
| 2 | **메타 정보** | 작성일, 작성자, 기밀등급 포함 여부 | ✅ |
| 3 | **정량적 근거** | 수치 주장에 데이터 출처가 있는가 | ✅ |
| 4 | **불확실성 표기** | 미확인/추정 정보에 표기가 있는가 | ✅ |
| 5 | **균형성** | 긍정/부정 양면이 균형 있게 기술되었는가 | ✅ |
| 6 | **논리 일관성** | 분석 내 모순이 없는가 | ✅ |
| 7 | **수치 정확성** | 보고서 내 수치 간 계산이 일치하는가 | ✅ |
| 8 | **한국어 품질** | 맞춤법, 전문용어 영문 병기 준수 | ✅ |
| 9 | **액션 지향** | 분석 결과가 구체적 액션으로 연결되는가 | 권장 |
| 10 | **분량 적정** | 과도하게 길거나 짧지 않은가 | 권장 |

### 1.2 보고서 유형별 추가 기준

**스크리닝 보고서:**
- [ ] 8개 평가 항목 모두 점수 기재
- [ ] 각 항목에 근거 1개 이상
- [ ] 총점 계산이 합산과 일치
- [ ] 강점/우려사항/확인사항 각 3개 이상

**IR 분석 보고서:**
- [ ] SWOT 4개 영역 모두 작성
- [ ] 핵심 질문 10개
- [ ] 투자 매력도 (1-5점) 명시
- [ ] 3개 에이전트 결과 교차 검증 반영

**리스크 경보:**
- [ ] 5축 리스크 매트릭스 완성
- [ ] 경보 수준 (긴급/주의/양호) 명확 분류
- [ ] 각 경보에 대한 대응 권고 포함
- [ ] 이전 경보와 비교 분석

**LP 보고서:**
- [ ] 기밀등급 "기밀" 표기
- [ ] 펀드 성과 지표 (IRR/MOIC/DPI) 포함
- [ ] 면책 조항 포함
- [ ] 민감 정보 (파이프라인 기업명 등) 미포함

**투자 검토 메모:**
- [ ] Investment Thesis 명확 (3-5문장)
- [ ] 리스크 매트릭스 (영향도/발생확률)
- [ ] 3가지 시나리오 (낙관/기본/비관)
- [ ] 투심위 논의 포인트 5개 이상

## 2. 자동 검증 규칙

### 2.1 수치 일관성 검증

보고서에서 자동으로 검증 가능한 항목:

```
검증 1: MRR × 12 = ARR (허용 오차 5%)
검증 2: 보유현금 / 월번레이트 ≈ 런웨이 (허용 오차 10%)
검증 3: 정량 평가 점수 합 = 60 이하
검증 4: 정성 평가 점수 합 = 40 이하
검증 5: 총점 = 정량 + 정성
검증 6: LTV / CAC = LTV/CAC 비율 (허용 오차 5%)
```

### 2.2 필수 섹션 존재 검증

각 보고서 유형별 필수 헤딩이 존재하는지 확인:

```
스크리닝: ["요약", "평가", "강점", "우려", "종합"]
IR분석:   ["Executive Summary", "SWOT", "리스크", "핵심 질문", "투자 매력도"]
리스크:   ["대시보드", "긴급", "주의", "액션"]
LP보고:   ["Executive Summary", "펀드 운용", "포트폴리오", "향후 계획"]
투자메모: ["Investment Thesis", "리스크", "시나리오", "논의 포인트"]
```

## 3. 피드백 루프

### 3.1 피드백 수집 프로세스

```
[1] 에이전트 → 초안 생성
     │
[2] 담당자 → 검토 및 수정
     │
[3] 수정 내용 기록
     │
     ├── 사실 오류 → data/reference/ 업데이트
     ├── 판단 오류 → rules/ 기준 보정
     ├── 형식 오류 → reporting-standards.md 보완
     ├── 누락 항목 → 에이전트 프롬프트 보강
     └── 톤/스타일 → 에이전트 지침 조정
```

### 3.2 피드백 기록 형식

`data/feedback/` 디렉토리에 피드백을 기록합니다:

```markdown
## 피드백 기록

| 일자 | 보고서 | 유형 | 피드백 내용 | 조치 | 반영 파일 |
|------|--------|------|-----------|------|----------|
| 2026-02-25 | aimedic_screening | 판단오류 | TAM 검증 부족 | 시장 검증 강화 | market-researcher.md |
```

### 3.3 주기적 품질 리뷰

| 주기 | 활동 | 목적 |
|------|------|------|
| 주간 | 생성 보고서 샘플 리뷰 | 즉각 품질 이슈 감지 |
| 월간 | 피드백 로그 분석 | 반복 패턴 파악 및 규칙 업데이트 |
| 분기 | 에이전트 프롬프트 전면 리뷰 | 대규모 개선 반영 |

## 4. 메모리 시스템 활용

### 4.1 에이전트 학습 패턴

에이전트의 프로젝트 메모리를 활용하여 분석 품질을 점진적으로 개선:

**축적할 정보:**
- 산업별 벤치마크 (실제 포트폴리오 기업 성과 기반)
- 자주 발견되는 리스크 패턴
- 성공한 투자의 공통 특성
- 실패/어려움을 겪은 투자의 패턴
- LP가 주로 질문하는 항목
- 투심위에서 자주 논의되는 쟁점

**메모리 업데이트 시점:**
- 신규 투자 확정 후
- 포트폴리오 기업 EXIT/실패 후
- LP 미팅 후 질문 패턴 업데이트
- 투심위 결과 반영 (통과/불통과 사유)
- 규제 환경 변화 시

### 4.2 메모리 파일 구조

```
.claude/projects/.../memory/
├── MEMORY.md              # 프로젝트 전체 요약
├── investment-patterns.md # 투자 패턴 및 교훈
├── industry-insights.md   # 산업별 인사이트
├── lp-feedback.md         # LP 피드백 및 질문 패턴
└── quality-log.md         # 품질 개선 이력
```

### 품질 KPI

| 지표 | 목표 | 측정 방법 |
|------|------|----------|
| 수정 비율 | < 20% | 담당자 수정 횟수 / 전체 보고서 |
| 사실 오류율 | < 5% | 팩트체크 실패 항목 수 |
| 재작성 비율 | < 5% | 전면 재작성 필요 보고서 비율 |
| 유용성 점수 | > 4/5 | 담당자 만족도 설문 |
| 시간 절감율 | > 60% | AI 활용 vs 수동 작성 시간 비교 |
